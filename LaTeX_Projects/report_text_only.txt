*{General Introduction}

When home appliances or hardware components break down, finding the right repair steps is usually a major headache. Nobody wants to flip through paper booklets or scroll endlessly in unorganized PDFs. Our team built PRO-WISE to solve this problem. Instead of manual search, users scan a product's unique QR code to open a tailored web dashboard on their phone. This workspace filters the catalog to show only relevant troubleshooting steps, compatible components, and a custom Gemini-powered chat assistant .

For manufacturers, running a call center or replying to support emails is slow and expensive. We built PRO-WISE to address this issue by providing immediate, context-aware assistance.

This document details the engineering lifecycle of PRO-WISE across seven chapters. Chapter 1 introduces the project context. Chapter 2 outlines system requirements and architecture. Chapters 3 to 7 describe the five iterative software releases, including authentication, catalog management, feedback collection, semantic AI search, and technician ticketing. The final conclusion reviews our achievements and plans future upgrades.

Preliminary Study

Industrial Context

Modern hardware products and machinery require fast, digital-first support channels. Customers want quick answers, but technical assistance remains fragmented. Most repair manuals are buried in static PDFs, legacy web portals, or physical paper guides. When a machine breaks down, users struggle to find simple, direct instructions.

Problem Statement

Traditional help-desk systems make support frustrating for users and expensive for brands. We identified four main pain points:

 Siloed Content: Repair manuals, video tutorials, and troubleshooting articles are hosted on separate websites, making search difficult.
 Dense Technical Books: Physical user guides are long and hard to navigate, pushing customers to ignore them and ask for human assistance immediately.
 Overloaded Call Centers: Because customers lack self-service tools, they call hotlines or schedule home visits for simple issues, driving up operating costs.
 Lack of Device Context: Support portals do not know which product model the user has, forcing them to manually search for serial numbers.

Analysis of Existing Solutions

Overview of Existing Systems

agraph{The iFixit Platform}
iFixit is a well-known community wiki for fixing gadgets and machines. It includes an AI assistant named FixBot that helps diagnose issues. However, since it is a public platform, it lacks the multi-tenant brand isolation and official approval workflows that companies require to control their support data.

agraph{The GE Appliances Support Portal}
GE Appliances offers digital manuals and technician bookings through its SmartHQ platform. While it includes AI chat tools for home diagnostics, the system is strictly proprietary. Other hardware manufacturers cannot use this ecosystem to support their own products.

Comparative Study

To place our project in the market, we compared these existing systems using five core functional metrics.

Comparative Evaluation and Positioning

Looking at Table , we see a clear gap in how product support is handled today. Community-driven platforms like iFixit have plenty of data, but they lack the privacy and brand control that enterprises require. On the other hand, proprietary systems like GE Appliances are highly secure but closed off, meaning other brands cannot use them. 

To bridge this gap, we built PRO-WISE as a multi-tenant, cloud-based SaaS platform. Here is how it stands out:

 Direct QR Scanning: Scanning a product QR code drops users straight into a customized troubleshooting space, skipping standard search engines entirely.
 Smart Moderation: We combine official developer-written guides with real-time AI screening to keep technical documentation safe and accurate.
 Built-in Gemini Assistant: We plugged in a RAG pipeline powered by the Google Gemini API so users can chat with their manuals in natural language.
 Connected Help Desk: When self-service troubleshooting fails, users can file a ticket to send the issue directly to a technician.

How PRO-WISE Works End-to-End

We designed PRO-WISE to link physical machines with smart digital services. Figure shows how users and administrators interact with the system components.

Here is the operational workflow step-by-step:

 Scanning the QR Code: A customer scans the product's QR sticker using their phone's camera web or mobile view.
 Extracting the ID: The app pulls the unique identifier from the QR code and requests the backend server via HTTPS.
 Data Fetching: The backend queries MongoDB for the product's details, PDF steps, and pre-computed text embeddings.
 Semantic Querying: If the user asks a question, the backend vectors the text, runs a similarity lookup in MongoDB, and passes the context to Google Gemini to get an exact answer.
 Rendering the App: The backend returns the guides, parts listing, and AI replies, showing a clean workspace to the user.
 Filing a Ticket: If the problem persists, the user submits a help request, which sends a notification and routes the ticket to available technicians.

Requirements Capture

Actors Identification

We defined five main actors who interact with the system:

 Client: Scans QR codes, views repair guides, watches troubleshooting videos, asks questions to the AI assistant, and leaves ratings.
 Technician: Edits guides, uploads photos or videos from site visits, handles ticket requests, and uses Gemini to draft manuals.
 Company Admin: Manages the brand's catalog (products, models), adds team members, and approves guide drafts before publication.
 Super Admin: Registers new manufacturing brands, updates system-wide configs, and inspects security audit logs.
 AI System: An automated background actor that computes text embeddings, ranks compatibility recommendations, and checks content quality.

Functional Requirements

We grouped the functional features into key modules. We designed each module to support multi-tenant isolation, ensuring brands cannot access each other's data :

 Authentication: Role-based permissions (Client, Technician, Company Admin, Super Admin) managed via JWT and Google login, enforcing brand isolation.
 Catalog Management: Administrative panels for company managers to handle products, manuals, and category links.
 Guides Editor: An interface for technicians to write step-by-step repair guides instead of just uploading static documents.
 Intelligent Search: Leveraging the Gemini API for semantic document queries, product recommendations, and language translation.
 Maintenance Ticketing: A portal for users to submit help requests and company managers to dispatch technicians.
 QR Routing: Generating barcodes linked to specific models that redirect users directly to help pages.

Non-Functional Requirements

To ensure the app is fast, secure, and reliable under heavy load, we met the following performance benchmarks:

 Security: Security was our top priority. We hashed all passwords using bcrypt with 10 salt rounds to block brute force attempts. For network security, we set up TLS 1.3 across all endpoints. We also protected the API against XSS and CSRF by storing stateless HS256-signed JWTs directly in HTTP-only, Secure, and SameSite=Strict cookies. It's clean and safe.
 Performance: We wanted the app to feel fast, aiming for sub-2-second loads. We audited pages with Lighthouse to optimize assets, hosting images on a CDN. In MongoDB, we avoided heavy offsets by using cursor-based pagination and created compound indexes on query-heavy fields to keep response times low.
 Responsive UI: We styled the frontend with Flexbox and CSS Grid so it adapts to any screen, keeping WCAG 2.1 AA guidelines in mind. On mobile, the React Native app uses viewport scaling, making sure buttons and text look good on any screen size from 4 to 6.7 inches.
 Uptime: To handle traffic spikes, we used Nginx as a reverse proxy and load balancer. We also set up MongoDB as a three-node replica set. If one database node goes down, the system handles the failover automatically without interrupting the user.
 Code Quality: We separated business logic into Sails.js services to keep controllers thin and clean. Code format is enforced using ESLint. On the client, we went with Redux Toolkit for state management, keeping everything type-safe with strict TypeScript.

 Interaction Steps:

 The client application initializes and loads the login page interface.

 The backend API responds with the token and user metadata.
 The client app saves the token, logs in the user, and redirects to the home dashboard.

 The user clicks the "Forgot Password" link on the interface.
 The client app displays the password recovery page.
 The user enters their registered email address.
 The app sends a POST request to the password recovery API route.
 The backend controller queries MongoDB to verify the email exists.
 MongoDB returns the matched user record.
 If the email is not registered, the server returns a 404 payload.
 If the email exists, the backend generates a secure recovery token.
 The server dispatches an email containing the recovery code.
 The client app displays the code verification form.
 The user enters the received code.
 The client app sends the code verification payload to the backend.
 The controller validates the code and checks its expiration time.
 If the code is invalid or expired, the UI displays an error alert.
 If valid, the server returns a temporary authentication state.
 The user types their new password.
 The backend hashes the new password with bcrypt and updates MongoDB.
 The database confirms the update and the user is redirected to login.

Comparative Study of AI Models

To choose the right model stack for our search and recommendation engine, we evaluated several vector embedding and generative systems.

Selection Criteria

We based our choice on three technical requirements:

 Spatial Accuracy: The vector space needs high dimensions to separate fine physical variations in hardware specifications.
 Language Support: The system must parse complex technical keywords in English, French, and Arabic.
 Infrastructure Cost: We wanted to minimize server costs (RAM, GPU) by using managed APIs rather than running heavy models locally.

Evaluated Models

We compared two main options:

 BAAI/bge-small-en : A local embedding model yielding 384-dimensional vectors. While easy to test locally, it only supports English and lacks the spatial precision we need.
 Google Gemini API : A managed cloud service that outputs 768/1536-dimensional vectors via text-embedding-004 and generates answers via gemini-1.5-flash. It supports multilingual queries, runs on serverless infrastructure, and provides high accuracy for hardware documentation.

RAG Architecture and System Integration

Semantic Search & RAG-Based Troubleshooting

Retrieval-Augmented Generation (RAG) merges vector search with large language models. When a user writes a troubleshooting question, the Sails.js backend fetches the closest guide steps from MongoDB using vector similarity, sending these chunks as context to the Gemini API to construct a clear reply.

Architecture Without RAG (Keyword-Based Approach)

In the traditional keyword approach, the user searches using text patterns. The backend performs a simple string match (like SQL LIKE or MongoDB \$regex) against titles and descriptions. The system returns a raw document list without analyzing query intent. Figure shows this basic structure.

This method misses related matches when users search with synonyms or typos. For instance, searching for "screen issue" returns nothing if the manual uses the term "display failure."

Architecture With RAG (Semantic Search Approach)

The RAG architecture implements a two-stage pipeline. During the indexing stage, we split manuals into small text paragraphs, convert each chunk into a 768-dimensional vector via the Gemini Embedding API, and store these arrays in MongoDB. During the query stage, the backend embeds the user's question, performs a cosine similarity lookup against database vectors, and passes the best matches to Gemini to generate the final response. Figure shows this workflow.

Hybrid Compatibility & Parts Recommendation

Our recommendation system uses a hybrid engine to suggest replacement components. To avoid recommending incompatible parts, we built a decision pipeline combining semantic vector similarity with strict engineering filters. Figure shows this workflow.

The pipeline executes the following stages:

 Initial Extraction: The backend parses the support ticket or user query to find the component name and target hardware model.
 Stage 1: Semantic Filtering: The system pulls inventory parts from MongoDB, gets their 768-D vectors from the Gemini API, and calculates cosine similarity against the query vector. We discard components scoring below a threshold.
 Stage 2: Deterministic Validation: The system checks the remaining components against three physical hardware constraints:

 Brand Matching: Ensures compatibility with the original manufacturer standards.
 Socket Fit: Confirms physical pins and dimensions align.
 Interface Validation: Verifies voltage and protocol signaling match.

 If any check fails, the part is filtered out.
 Sorted Output: We sort compatible parts by similarity score and display them as certified replacements.

Technical Challenges and Solutions

Integrating external AI services introduced two main challenges:

 API Rate Limits: The Google Gemini API restricts query rates. We solved this by implementing an in-memory queue with exponential backoff on retries.
 Token Limits: Long manuals consume context windows. We resolved this by pre-segmenting guides into text chunks of maximum 1000 characters before embedding.

*{General Conclusion and Perspectives}

*{General Conclusion}

Our team developed PRO-WISE to streamline appliance technical support and coordinate on-site repairs. We structured the project as a monorepo that connects manufacturers, field technicians, and customers into one cohesive, context-aware environment.

To build the platform modularly and systematically, we split the implementation into five successive sprints:

 Core Security: We implemented role-based access control and stateless token authentication to secure backend API routes from the start.
 Content Management: We designed category catalogs, markdown manuals, and interactive guide editors for brand administrators.
 Engagement and Logs: We built feedback review forms, real-time WebSocket notifications, and background database audit logs.
 Intelligent Layer: We integrated the Google Gemini API for vector semantic search, automated compatibility matching, and real-time community content moderation.
 Field Support: We created a full-lifecycle support ticketing flow and dedicated mobile jobs dashboards for repair technicians.

Running a Sails.js backend separately from the React web dashboard and React Native mobile app allows us to host and scale services independently. Comparing PRO-WISE with existing systems like iFixit and GE Appliances highlights how linking direct QR code scans to vector semantic search models modernizes technical maintenance.
